BITLOCKER

BitLocker es una herramienta de cifrado desarrollada por Microsoft para sistemas operativos Windows, principalmente destinada a proteger la información almacenada en dispositivos como discos duros y unidades extraíbles. Su propósito principal es evitar que personas no autorizadas puedan acceder a la información contenida en un equipo si este se pierde, es robado o alguien intenta extraer el disco para leerlo en otro sistema.

¿Qué es BitLocker y cómo funciona?

BitLocker funciona cifrando completamente el volumen de almacenamiento de una unidad (por ejemplo, la unidad C: donde está instalado el sistema operativo), usando un algoritmo de cifrado como AES (Advanced Encryption Standard). Este cifrado convierte los datos almacenados en un formato ilegible para cualquier persona sin la clave o autenticación necesaria para descifrar la unidad.

Componentes de BitLocker

  1. Clave de cifrado de volumen (VEK): Es la clave principal que BitLocker utiliza para cifrar los datos. Esta clave es única para cada volumen cifrado.
  1. Protección de la clave (Key Protectors): Son mecanismos para proteger la clave de cifrado y asegurar el acceso autorizado al volumen. Existen varias formas de protección que pueden utilizarse individualmente o en combinación:
    • TPM (Trusted Platform Module): Un chip de seguridad que guarda las claves de cifrado y verifica la integridad del sistema antes de permitir el descifrado.
    • PIN: El usuario puede configurar un PIN que debe ser introducido junto con el TPM para desbloquear el sistema.
    • Contraseña o clave de recuperación: Una clave de 48 dígitos que puede usarse para desbloquear la unidad en situaciones de emergencia.
    • Unidad USB: Es posible almacenar la clave de desbloqueo en una unidad USB que debe estar presente al arrancar el sistema.

NTLM (NT LAN Manager)

NTLM es un protocolo basado en el mecanismo de Desafío-Respuesta (Challenge-Response). No utiliza un servidor de tickets centralizado en cada interacción directa; en su lugar, el cliente debe demostrar al servidor que conoce la contraseña sin enviarla en texto plano.

Flujo de Funcionamiento (El "Apretón de Manos")

Cuando un usuario quiere acceder a un recurso en un servidor web, una carpeta compartida (SMB) o una base de datos usando NTLM, ocurre lo siguiente:

  1. Solicitud de Negociación (Negotiate): El cliente envía una petición al servidor indicando que quiere autenticarse y detalla las versiones de NTLM que soporta.
  1. El Desafío (Challenge): El servidor genera un número aleatorio de 8 bytes (llamado Nonce o desafío) y se lo envía al cliente.
  1. La Respuesta (Response): El cliente toma el hash de la contraseña del usuario (el hash NT), cifra el Nonce recibido utilizando ese hash como clave, y envía el resultado (la respuesta) de vuelta al servidor.
  1. Validación (Passthrough Authentication): * Si el servidor es local, compara el resultado con su propia base de datos SAM.
    • Si es un entorno de dominio (Active Directory), el servidor no conoce el hash del usuario. Por lo tanto, el servidor envía el Challenge y la Response al Controlador de Dominio (DC) a través de un canal seguro (Netlogon).
  1. Veredicto: El DC descifra la respuesta usando el hash del usuario que tiene guardado, verifica si coincide y le dice al servidor si el usuario es legítimo o no.
⚠️ El gran problema de NTLM: El servidor intermedio recibe la respuesta cifrada del cliente. Si un atacante logra posicionarse en el medio (MitM), puede capturar esa respuesta y reutilizarla en otro servidor que acepte el mismo hash (Ataques de NTLM Relay), o intentar romperla por fuerza bruta de forma offline.

Kerberos

Kerberos es un protocolo de autenticación de red basado en Tickets y criptografía de clave simétrica. Introduce a un tercero de confianza: el KDC (Key Distribution Center), que en entornos Windows corre dentro del Controlador de Dominio.

Kerberos asume que las redes no son seguras y evita por completo enviar hashes de contraseñas a través de la red durante el acceso a los servicios.

Componentes Clave del KDC

  • AS (Authentication Service): Autentica inicialmente al usuario.
  • TGS (Ticket Granting Service): Emite los tickets para los servicios específicos (como un servidor de archivos o SQL).

Flujo de Funcionamiento (Las 3 Fases)

Fase 1: Obtención del TGT (Ticket Granting Ticket)

  1. Solicitud de AS: El cliente envía una petición al AS del KDC. Para demostrar quién es, cifra una marca de tiempo (timestamp) usando el hash de su contraseña.
  1. Emisión del TGT: El AS descifra la marca de tiempo (si funciona, la contraseña es correcta). El AS responde enviando el TGT (cifrado con la clave secreta del KDC, por lo que el cliente no puede alterarlo) y una Clave de Sesión de la AS.

Fase 2: Solicitud de acceso a un servicio

  1. Petición al TGS: El usuario quiere entrar a un servidor de archivos. Toma su TGT, añade un autenticador y los envía al TGS, solicitando acceso al servicio específico (identificado por su SPN - Service Principal Name).
  1. Emisión del Ticket de Servicio: El TGS valida el TGT, genera un Ticket de Servicio (ST) cifrado con la clave secreta del servidor de destino, y se lo entrega al cliente.

Fase 3: Autenticación ante el servidor

  1. Presentación del Ticket: El cliente envía el Ticket de Servicio directamente al servidor de archivos.
  1. Acceso Concedido: El servidor descifra el ticket con su propia clave, verifica la identidad del usuario y le da acceso. ¡El servidor nunca tuvo que hablar con el Controlador de Dominio para validar al usuario en este momento!

Editamos la nueva GPO que creamos

Le damos a Computer Configuration > Administrative Templates > Windows Components > Bitlocker Drive Encryption y en esta ventana vamos a hacer la magia.

Store Bitlocker recovery information in AD DS

Esta política sirve para guardar automáticamente la clave de recuperación de BitLocker en tu servidor de Active Directory (AD DS) cuando un usuario cifra su disco.

Configure use of hardware-based encryption for fixed data drives

Esta política sirve para decidir si BitLocker debe usar el chip de cifrado integrado del propio disco duro (cifrado por hardware) o si debe realizar el cifrado usando el procesador de la computadora (cifrado por software) en las unidades de datos fijas (discos duros internos secundarios).

Allow network unlock at startup

Esta política, llamada "Allow network unlock at startup" (Permitir desbloqueo de red al iniciar), sirve para desbloquear automáticamente las computadoras con BitLocker cuando se encienden dentro de la red cableada de la empresa, sin necesidad de que el usuario escriba un PIN.

Enforce drive encryption type on operating system drives

Esta política, llamada "Enforce drive encryption type on operating system drives", sirve para definir si BitLocker debe cifrar todo el almacenamiento del disco duro completo o solamente el espacio que tiene datos utilizados, bloqueando o permitiendo la decisión al usuario final.

Choose how BitLocker-protected operating system drives can be recovered

Esta política, llamada "Choose how BitLocker-protected operating system drives can be recovered", sirve para controlar qué opciones de recuperación tiene un usuario si pierde su PIN o llave de inicio, y cómo se respalda esa información de emergencia.

Require additional authentication at startup

Esta política, llamada "Require additional authentication at startup" (Requerir autenticación adicional al iniciar), sirve para definir las opciones obligatorias que pide la computadora justo al encenderse, controlando si se utiliza solo el chip TPM o si se añaden capas extras como el PIN.

Esta política esta deshabilitada por que el pin lo vamos a poner cuando ya tengamos el disco cifrado.
